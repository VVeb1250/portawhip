# Writer Consolidation Plan (for Codex)

> Evidence-backed plan to move Portawhip from "many backends writing the same
> host files" to "one writer per lane + one reconciler." Grounded in a live
> 3-way head-to-head, not docs. Follow the VISION principles: **delegate don't
> rebuild**, **live-probe never overclaim**, **no daemon until a proven gap**,
> **single writer per target file**.

## 0. Evidence (live 3-way head-to-head, 2026-07-13)

Harness: `docs/writer-consolidation-harness.sh` (project-scope, isolated temp dirs, real
`~/.claude.json` guarded — zero test servers leaked into it). Two http MCP
servers (`deepwiki`, `context7`) written via each backend, then a drift test on
one shared codex target.

| Backend | wrote 2 servers | idempotent | codex MCP encoding | ownership model | preview | backup |
|---|---|---|---|---|---|---|
| **add-mcp** | ✅ claude `.mcp.json` + codex TOML, both hosts | ✅ | `type = "http"` | **surgical merge** (only mcpServers touched) | — | non-destructive merge |
| **ai-config-sync-manager** | ✅ → codex | ✅ | `transport = "streamable_http"` inside `# BEGIN/END ai-config-sync mcp-servers` marker block | **marker-block merge** (coexists w/ human edits) | ✅ excellent `--dry-run --plan-json` | ✅ `~/.ai-config-sync-manager/backups` |
| **@agents-dev/cli** | materialized **11 hosts** from `.agents/agents.json` | ✅ | full TOML regen | **total-ownership**: `# Generated by agents sync. Do not edit manually.` — rewrites whole file, injects default servers (fetch/filesystem/git) on init | `sync --check` | `agents reset` |

**Three backends produce three different codex MCP encodings.** That alone
guarantees drift if more than one writes codex.

### Decisive result — DRIFT WAR: OSCILLATES (proven)

On one shared codex target:
1. `ai-config-sync` writes its marker block (2 http servers) → hash A.
2. `agents connect` **clobbers the entire file** ("do not edit manually") —
   ai-config-sync's servers are gone, replaced by agents' defaults → hash B.
3. `ai-config-sync` re-run re-injects its block → hash C.

Hashes never converge. **Two steady-state writers on one target = permanent
fight.** The Codex report's central thesis is now empirically confirmed.

### Open risk found during the test

`agents mcp add <url> --name X` did **not** persist my servers — `agents mcp
list` showed only the init defaults (fetch/filesystem/git). Before trusting
@agents-dev as THE MCP writer, its MCP-ingestion path must be verified (see
Phase 1). Everything else about @agents-dev (11-host fan-out, idempotency,
reset) checked out.

## 1. Decision (what the evidence dictates)

> **UPDATED 2026-07-13 after the §1d gates all passed. The original pick
> (`@agents-dev/cli` as core) is superseded by Fork A: `rulesync` as sole writer
> for both scopes.** `@agents-dev` was project-only and could never cover global;
> `rulesync` matches it on project and adds global. Rationale preserved in §1d.

- **One writer per lane/scope. No exceptions.** Overlapping steady-state writers
  fight — proven (§0). The disease is specifically **total-ownership writers**
  (§1c); surgical writers coexist.
- **Sole fan-out writer = `rulesync`** (both scopes). Canonical `.rulesync/` →
  generated host files. Portawhip owns `.rulesync/` content (strips the scaffold
  defaults) and treats every host file as generated output.
- **`ai-config-sync-manager` → migration/import only** (never steady-state).
- **`add-mcp` → discovery/import (read side) only — but not droppable.** Its
  `listInstalledServers()` unions MCP servers across all detected hosts, which
  `rulesync import` cannot do (see §1e). Only its *write* role is retired.
- **`@agents-dev/cli` → retire** (project-only, fiddly MCP ingestion, injects
  default servers — rulesync dominates it on every tested axis).
- **`mise` → keep** (CLI tools; no host-config overlap).
- **`agent-skill-manager` → long-tail skill hosts only** (hosts rulesync's
  `--targets` doesn't cover).

## 1b. Scope model (project vs global) — verified gap

The head-to-head was **project-scope only** (deliberate, for safety). Global
scope is a different problem and is **not yet handled**. Verified fact:
**`@agents-dev/cli` has no global/user scope** — only `--path <dir>` (a project
dir). It materializes into a project's `.claude/`, `.codex/`, … — never
`~/.claude.json`. So "one core writer for both scopes" is false.

| | Project scope | Global scope |
|---|---|---|
| blast radius | low — files in a repo, disposable, git-diffable | **high — writes the live config the running session reads** |
| canonical lives | `.agents/` **in the repo** (committed, team-shared) | must live under `~` (e.g. `~/.config/portawhip/`) — owner, backup, migration all TBD |
| broad fan-out writer | ✅ `@agents-dev/cli` (11 hosts) | ❌ **none exists** |
| who can write it | @agents-dev | `add-mcp -g` (MCP only) · `ai-config-sync --scope global` (claude↔codex) · `asm` (skills → provider dirs) |
| gate | preview → apply | **mandatory backup + never auto + explicit confirm every time** |

**Consequence: writer-per-lane must split into writer-per-lane-per-scope.**
- **Project = the primary, safe, team-shareable lane.** Canonical `.agents/` in
  the repo; `@agents-dev/cli` fans out. Focus here first.
- **Global = the personal, per-machine, dangerous lane.** No broad writer;
  compose `add-mcp -g` + `ai-config-sync --scope global` + `asm` + portawhip
  adapters for gaps. Canonical under `~`. Higher blast radius → stricter gates.

Two unresolved design decisions:
1. **Cross-scope precedence/dedup.** Hosts *merge* global + project (Claude
   reads `~/.claude.json` mcpServers **and** project `.mcp.json`). Writing the
   same server in both scopes double-declares it. Rule needed: a server declared
   at global scope must not be re-emitted at project scope (or respect the
   host's own precedence).
2. **Global drift war is untested.** Project drift is proven (§0). On global,
   `add-mcp -g` and `ai-config-sync --scope global` both can write
   `~/.codex/config.toml` with different encodings — likely fights, unproven.
   `@agents-dev` is absent from the global fight (can't write global), so global
   has fewer competing writers; overlap is MCP only. **Must be tested with full
   backup→test→restore before any global apply — never on the live config
   unguarded.**

## 1b-solved. Cross-scope dedup by DERIVED scope, not a hand-kept list (implemented 2026-07-13)

§1b.1 left an open question: which MCP servers are global vs project, so a server
isn't declared in both scopes? Hand-classifying each server (harness-router →
project, gortex → ?, context7 → global) is **hardcoded decision logic** — it
breaks the moment a user loads their own tool, and violates VISION's "hardcode
data, never decision logic." Solved instead by **deriving scope from observable
config properties** — same "detection, not a list" stance as `isHostPrivateMcp`.

**`core/surface/scope-derive.mjs` (built + tested, 10/10):**
- `deriveScope(config, { discoveredGlobal, projectRoot })` →
  1. **config is project-bound** (any command/arg/cwd is a repo-relative path:
     `.`/`./`/`../`, a `${workspaceFolder}` placeholder, a `*.mjs|py|...` script
     file, or an absolute path *inside* `projectRoot`) → **forced `project`** (a
     repo path is meaningless in the user's global config);
  2. else **portable** (bare package / public URL) → takes the scope it was
     **discovered at** (`listInstalledServers({global})` already gives this);
  3. else → default `project` (never pollute the user's global by guessing).
- Conservative on false positives: scoped npm names (`@scope/pkg`) and absolute
  paths outside the repo stay portable.
- **Verified on the real reported conflict:** harness-router's installed config
  `node E:\portable-harness-v2\server\mcp-server.mjs` → `project` (reason:
  "project-bound path"). Its "differs across 6 hosts" conflict disappears because
  it is never fanned out to global at all.

**Single-emission invariant (the dedup):** each canonical server carries its
derived scope as *data*; the reconciler emits it in exactly that one scope and
relies on host-native merge (project inherits global) — so the same server is
never declared twice. Extend Codex's `writer-invariant`/`ownership-ledger` with
"no server appears in >1 scope."

**Wiring (for Codex — not yet applied, it changes seed behavior):**
- `seedMcpCanonical` (rulesync-canonical.mjs): when `scope === "global"`, drop
  servers whose `deriveScope` is `project`; when `scope === "project"`, drop
  servers already emitted globally (host merge covers them). Pass `projectRoot`
  and per-server `discoveredGlobal` (from a `{global:true}` vs `{global:false}`
  pass) into `deriveScope`.
- Curated `recipe.yaml` `scope:` stays valid as an **optional override**;
  default is inferred, so no per-tool list is maintained.

**gortex conflict = a migration artifact, not a steady-state scope question.**
`claude ≠ codex` is content drift from the pre-consolidation multi-writer era.
Codex already made import surface conflicts instead of silently picking the first;
the user resolves it **once**, the canonical stores the choice, and after that a
single `.rulesync/` source makes divergence structurally impossible. It does not
need a permanent hand-picked rule.

**Future user tools:** a newly loaded tool flows through the identical pipeline —
discover → `deriveScope` → single-scope emit — with zero per-tool code or list.
That is the whole point: the classification is a function of the tool's own
config, not a map someone has to maintain.

## 1c. Global drift test result + the sharper rule (tested 2026-07-13)

Ran the global drift test in a **HOME-overridden fake home** (real
`~/.codex/config.toml` + `~/.claude.json` hashed before/after = UNCHANGED_OK;
zero real writes). Result:

- `add-mcp -g` + `ai-config-sync --scope global` on the same global codex target
  **coexisted (STABLE)** — ai-config-sync saw the servers already present and
  no-op'd. Both are **surgical mergers**.
- Contrast with project scope, where `@agents-dev` clobbered everything.

**Sharper rule the evidence supports:** the drift disease is **total-ownership
writers** (`@agents-dev` regenerates the whole file: "do not edit manually").
**Surgical writers coexist**; a total-ownership writer poisons any shared file.
So the rule is not merely "one writer per file" — it is:
> Never mix a total-ownership writer with any other writer on a file. Surgical
> writers may share a file only if they own disjoint regions/capabilities and
> use the same encoding.

`@agents-dev` can't write global anyway, so **global is inherently less
drift-prone than project** — its only writers are surgical.

Caveat (untested): ai-config-first ordering, and the encoding mismatch
(`type="http"` vs `transport="streamable_http"`) if both write MCP. Resolve by
giving global-MCP to exactly one writer.

## 1d. Global-lane candidate: `rulesync` (tested, strong)

`rulesync` was evaluated live in the fake home. It is the first tool found that
**covers both scopes and the full capability set in one canonical source**
(`.rulesync/`):

- `generate -g` writes **global/user-scope** files (the gap `@agents-dev` can't
  fill) — verified writing fake `~/.claude.json` + `~/.codex/config.toml`.
- `--targets *` (12 hosts), `--features rules,ignore,mcp,subagents,commands,skills,hooks,permissions`
  (broader than `@agents-dev`), `--dry-run`, `--check`, `import` (migration).
- **Idempotent:** `generate --check` reports clean after a generate.
- **Surgical on `~/.claude.json`:** seeded the file with `projects` + `userID` +
  `mcpServers`; after `generate -g --features mcp` the non-MCP state (`projects`,
  `userID`) **survived** — only `mcpServers` was rewritten. This is the critical
  safety property for a global writer (`.claude.json` holds all Claude Code
  state; a total-ownership writer there would nuke it).
- **Coexists with `add-mcp`:** after `add-mcp -g` wrote the same codex target,

**Implication:** `rulesync` could **replace `@agents-dev` (project) *and* the
composed global stack (`add-mcp` + `ai-config-sync` + `asm`)** with one
canonical source spanning both scopes — collapsing the two-lane complexity.

## 1e. Migration reality: `rulesync import` is per-host, not multi-host union (tested)

Tested `rulesync import` 2026-07-13. It imports **one target at a time and
overwrites `.rulesync/mcp.json` per import — it does not union across hosts.**
Seeded a project with claude MCP (`deepwiki`, `localstdio`) + codex MCP
(`codexonly`); `import --targets claudecode` then `import --targets codexcli`
left `.rulesync/mcp.json` with **only `codexonly`** — the claude servers were
overwritten, not merged.

Consequence for retiring the old backends:
- **`ai-config-sync-manager` → fully droppable.** Its claude↔codex migration
  niche is one-time and subsumed once `.rulesync/` is canonical.
- **`add-mcp` read side → keep (not droppable).** Portawhip's `discover.mjs`
  already unions installed MCP across all detected hosts via
  `add-mcp.listInstalledServers()` — the exact multi-host union `rulesync
  import` lacks. Only add-mcp's *write* role retires.

**Migration flow:** `discover.mjs` (add-mcp multi-host union) → write
`.rulesync/` canonical once → `rulesync generate` owns steady-state from there.

## 1f. The two capabilities `rulesync` does NOT cover — strategy: contribute, don't add a dep

`rulesync --features` has no `plugins`, and neither it nor Portawhip has a write
lane for **hooks embedded inside skills/plugins**. Decision:

- **Do NOT add a third sync dependency for these.** A separate plugin/embedded-hook
  writer would touch dirs `rulesync` already owns → reintroduces the exact
  total-ownership drift disease this whole plan eliminates (§1c). Non-negotiable.
- **Default = defer (inventory/route only).** Host plugin formats are still
  moving (Cursor 2.5 just shipped, Codex plugin partial, VS Code agent plugins
  in preview). Per VISION "proven-gap before scope," do not compile a moving
  target. Inventory and route these; don't sync them until a real user needs it.
- **When a real need appears → contribute the feature to `rulesync`, not a new
  dep.** rulesync already has the 12-host adapter framework, so a `plugins`
  feature / embedded-hook extraction is incremental there and keeps the
  single-writer architecture intact. Keep a thin local Portawhip adapter as a
  fallback (writing to rulesync-canonical or a non-overlapping lane) only until
  the upstream PR merges. Bonus: an upstream PR to an actively-used OSS tool is a
  stronger artifact than a niche in-repo lane. Risk to manage: rulesync is
  single-maintainer — the local fallback covers merge-timeline uncertainty.

## 2. Phases (sequenced; each has a test-before-adopt gate)

### Phase 1 — Adopt `rulesync` as sole writer; migrate off the rest
Decision is locked (Fork A, §1d gates all passed). This phase executes it.
- Add `rulesync` as a dependency. Build a thin adapter that owns `.rulesync/`
  content generated from Portawhip's canonical (strip the `init` scaffold
  defaults — serena/context7/sample rules — so nothing opinionated ships).
- Rewrite `core/surface/config-sync-backends.mjs`: `rulesync` is the single
  steady-state writer (`generate` for apply, `--dry-run`/`--check` for
  preview/status, both scopes via `-g`). Demote `ai-config-sync` to a
  `migrate`-only action; demote `add-mcp` to discovery/import; remove
  `@agents-dev/cli` from the steady-state path.
- Migration (per §1e): seed `.rulesync/` via `discover.mjs`'s multi-host union
  (`add-mcp.listInstalledServers()`), NOT `rulesync import` alone — import is
  per-host and overwrites, so it drops servers that live on only some hosts.
  `rulesync import` is fine only as a single-host bootstrap.
- **Acceptance:** exactly one steady-state writer (`rulesync`); a doctor line
  asserts no lane lists a second; `generate --check` is clean after an apply on
  both scopes.

### Phase 2 — Ownership ledger + strip injected defaults
- Add a per-target-file **ownership ledger** (`{ path → owning writer + content
  hash }`). Any write by a non-owner, or a hash that doesn't match the last
  generated output, is flagged as drift, not silently overwritten.
- Neutralize @agents-dev's init defaults: portawhip manages `agents.json`
  content and removes fetch/filesystem/git unless the user actually declared
  them.
- **Acceptance:** running the core writer twice = zero diff; a hand edit to a
  generated file is *detected* (reported), not clobbered.

### Phase 3 — Refresh host adapter catalog (LIVE-PROBE, not docs)
- `core/surface/hook-targets.mjs:38` is behind. The Codex research report claims
  Codex now has SessionStart, Cursor/Copilot/VS Code have hooks, Gemini has
  commands/agents. **Those claims are doc-sourced — do not flip the catalog on
  faith.** Live-probe each host on this machine first (write a throwaway hook,
  confirm it fires), exactly as VISION requires.
- Only flip a lane from `unsupported` → supported after a live probe passes.
  Where a probe can't run (host not installed), keep `unsupported` and note
  "unverified," not "supported per docs."
- **Acceptance:** every catalog entry has a live-probe result behind it; none
  rely on documentation alone.

### Phase 4 — Single manual reconciler (the real deliverable)
- `portawhip sync check | apply | verify` — one command covering MCP, CLI,
  skills, agents, instructions, hooks, local plugin packages.
- Must do: backup → atomic write → rollback-on-error → runtime smoke test per
  detected host. Reuse ai-config-sync's dry-run/plan-json shape as the preview
  model (it was the best of the three).
- **Scope-aware:** `--scope project|global|all`. Project uses the in-repo
  `.agents/` canonical + @agents-dev. Global uses the composed writers
  (`add-mcp -g`, `ai-config-sync --scope global`, asm) + a `~`-based canonical,
  with a mandatory backup and explicit confirm — global apply is never implied
  by an `all` run without a second confirm. Enforce the cross-scope dedup rule
  (§1b.1) so a server isn't declared in both scopes.
- **Gate:** one full manual reconcile must pass on real hosts (with backup)
  before any automation is wired — run project scope first, then global only
  after a backup→test→restore dry run proves the global drift behaviour (§1b.2).
  This is the "prove the manual path" gate the report and VISION both require.

### Phase 5 — Multi-host SessionStart trigger (only after Phase 4 passes)
- Do not tie auto-sync to Claude alone. Codex/Copilot session hooks (once
  Phase-3-verified) call the **same reconciler**, behind a lock + throttle so
  concurrent hosts don't stampede.
- **Acceptance:** two hosts starting near-simultaneously produce exactly one
  reconcile (lock proven), and it's the same reconciler as the manual path.

**DONE (2026-07-14) — with two real findings along the way, not assumed:**
The worker (`scripts/sync/auto-sync.mjs`: lock+throttle+log, calls
`reconcile.mjs`) and the fire-and-forget SessionStart trigger
(`adapters/hooks/universal-hook.mjs`) already existed from an earlier phase,
but were dead: `router.config.yaml` had no `autoSync` key (default
`enabled:false`) AND — the real finding — **no host had the SessionStart hook
actually installed**, because:

1. **rulesync's `hooks` feature is non-functional for `claudecode`** (live-
   tested, not doc-trusted): 3 input schemas tried against
   `.rulesync/hooks.json` (flat `{type,command}`, nested `{matcher,hooks}`,
   both under the zod-required `Record<event, Array<...>>` shape) — every one
   generated an **empty** `.claude/settings.json` hooks key. Undocumented
   format, unclear if a genuine bug or an unsupported target. `link-hooks.mjs`
   (the actual working writer) had been gated to `status`-only, throwing
   "read-only; use portawhip sync apply so Rulesync owns the write" — an
   assumption this test disproves.
   - **Fix:** dropped `"hooks"` from `rulesync.jsonc`'s features (rulesync no
     longer touches the key at all) and restored `link-hooks.mjs` install/
     remove as the sole writer for that key — a **surgical writer on a
     disjoint region** of a file rulesync otherwise owns (mcp/permissions/
     etc.), exactly the coexistence rule §1c already established. Added
     `npm run hooks:install` / `hooks:remove` (previously only `hooks` =
     status existed — install had no script at all).
2. **Ownership ledger is whole-file-hash, not per-key**, so once link-hooks
   wrote the `hooks` key, `sync check` reported false-positive drift on
   `.claude/settings.json` / `.codex/hooks.json` (rulesync's stored hash no
   longer matched the file rulesync + link-hooks now jointly produce). Fixed
   with `refreshLedgerBaseline()` in `link-hooks.mjs`: after a real
   install/remove, re-claims the path under the **same writer** rulesync
   already holds it under, with the new combined hash — same pattern
   `reconcile.mjs`'s own `recordOwnership` uses after its apply. This is a
   point fix, not a general per-region ledger; if a third disjoint-region
   writer ever appears on a rulesync-owned file, this pattern generalizes but
   isn't automatic.
3. Live-verified end to end: `npm run hooks:install` → claude-code/codex/
   gemini-cli all `linked`, idempotent on re-run (`no-op`). Fired the actual
   installed command (`node adapters/hooks/universal-hook.mjs --event
   session_start --host claude-code`) exactly as a real session start would —
   `auto-sync.log` recorded `fan-out ok`, state updated, immediate second fire
   correctly reported `{"skipped":"throttled"}`. `sync check --scope project`
   clean (`status: success, drift: []`) after all of it. Full suite 251/251.
4. `router.config.yaml`: `autoSync: { enabled: true, throttleMinutes: 60 }`
   (was `false` by default — flipped, since the manual-path gate above and
   Phase 4's project-scope proof (43 targets, 0 drift, pre-dating this) were
   already satisfied). **Global stays out of auto-sync** — no automated apply
   there yet, matching §1b.2's stricter gate.

### Phase 6 — Watchdog (last; only if a proven gap remains after Phase 5)

**NOT STARTED (2026-07-14) — gate not met, on purpose.** Phase 5 (SessionStart
auto-sync) only just went live above; it hasn't run across real multi-day
usage yet, so there is no evidence of a gap a watchdog would close. Building a
timer/FS-watch daemon now would be scope-adding on faith, the exact thing this
plan's own guardrails (§3) forbid. `scripts/sync/sync-surfaces.test.mjs` has a
standing regression test — "exposes manual sync/check only (no watchdog
command)" — that will need to be deliberately updated, not just left to fail,
when this phase is actually taken up.
- Watch **canonical files only**, one-way fan-out, debounce, single-instance
  lock, hash ledger to separate user edits from generated output, periodic full
  reconcile as a backstop for missed FS events. Never sync secrets. Report drift
  on generated-file edits; name hosts that need a session reload.
- **Gate:** only build this if SessionStart-triggered reconcile (Phase 5) leaves
  a demonstrated gap. Per VISION: no daemon until the gap is proven, not assumed.

## 3. Do-not-do (guardrails from the evidence)

- Do **not** run `ai-config-sync` and `@agents-dev` as steady-state writers on
  any shared target — proven to oscillate.
- Do **not** flip host-support lanes from documentation — live-probe.
- Do **not** own a canonical compile schema for a capability where an existing
  tool already compiles it well, unless a probe shows a real gap (delegate first).
- Do **not** open the watchdog before the manual reconciler + ownership ledger
  are proven — multiple writers + a watchdog = the oscillation above, on a timer.
